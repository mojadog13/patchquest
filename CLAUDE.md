# Patch Quest — 減算合成を体で覚えるお題クリア型シンセ入門アプリ

> シンセサイザー未経験の大人が、「この音を作れ」というお題をノブ操作でクリアしながら減算合成の仕組みを学ぶWebアプリ(PWA)。レッスンの進行とともにシンセのモジュールが解放されていく、つまり「シンセの信号経路そのものが進行度システム」であることがコンセプトの核。

## 背景・文脈

シンセサイザーに興味を持っても、VCO・VCF・ADSRといった用語の壁と「ノブが多すぎて何をすればいいかわからない」問題で挫折する人が多い。既存のシンセアプリは実機のエミュレーションが中心で、学習体験として設計されたものは少ない。

本プロジェクトは「AIと共に作りたい個人アプリ」の第一弾。開発者(J)は制御工学(モデルベース制御)が専門であり、信号経路・伝達特性・パラメータ同定の考え方をアプリ設計(特に判定器の設計)に活かす。開発はClaude Codeを用いて行う。

## コンセプトとコアループ

Wordleやピクロスのように「1問数分で完結する音作りパズル」として設計する。

1. お題の音を聴く(例:「霧笛のような低い音」)
2. 現在アンロック済みのノブを操作して音を作る
3. 自分の音とお題を聴き比べ、提出する
4. 判定 → クリアで次のレッスンへ。新モジュール解放時は短い解説カードを表示

1問につき学ぶ概念は1つに絞る。これが挫折させないための最重要原則。

## カリキュラム設計(=モジュール解放順)

減算合成の信号経路(オシレータ → フィルタ → アンプ)をそのまま章立てにする。各章5〜8問、全体で30問前後を想定。

- **第1章 オシレータ**: 波形(サイン→矩形→ノコギリ)とピッチのみ。「波形で音色が変わる」を体感する
- **第2章 フィルタ**: LPF(ローパスフィルタ)のカットオフとレゾナンス。「こもる/開く」「ミョンミョン」の正体を知る
- **第3章 エンベロープ**: ADSR。「ピアノのような減衰」「弦のような立ち上がり」を作る
- **第4章 LFO**: ビブラート、ワウ、トレモロ。「揺れ」の付け方を学ぶ
- **第5章 統合**: 全モジュールを使った実用音色の再現(ベース、リード、パッド、効果音)
- **卒業制作**: 自由制作+パッチのシェア(URLにパッチをエンコードして共有)

## UI設計方針

- スマホ縦持ちを基準とする。鍵盤は置かない
- **ラック型レイアウト**: モジュール(オシレータ→フィルタ→アンプ/エンベロープ→LFO)を実機風パネルとして信号の流れ順に縦に並べる。各パネルが自分のノブを内包する。抽象的なブロック図は別に描かず、**パネルの並びそのものが信号経路図**として機能する
- 未解放モジュールは彩度を落とした状態でラック内に鎮座させる(存在は見えるが触れない)。レッスン進行でパネルに色が灯っていくことが、カリキュラム進行の可視化を兼ねる
- 操作中のモジュールはハイライト(縁の発光など)で示す
- 上部固定ヘッダー: アプリ名、レベルメーター、マスターボリューム、お題再生/自分の音再生の切替
- 発音は大きな再生ボタンによる単音トリガー。必要な問題のみピッチスライダーを追加する

## ビジュアルデザイン方針

参照デザイン: ブラウザ版アンプシミュレータ「Valve Stack」のUI(スクリーン録画あり)。ユーザーが特に評価しているのは「ノブやパネルの質感」と「画面レイアウト構成」の2点であり、この2点を忠実に踏襲する。

**質感(スキュアモーフィック)**
- ダークチャコールの背景に、実機のギターペダル/ラック機材を模したパネルを配置する
- モジュールごとに彩度の高い署名色を割り当てる(例: オシレータ=赤系、フィルタ=青系、エンベロープ=緑系、LFO=紫系。最終配色は実装時に調整)
- ノブは立体感のある黒い円形(CSSグラデーション+box-shadow、またはSVGで描画。画像アセットは使わない)。回転位置インジケータと、直下に小さな値表示を持つ
- パネル上のラベルは小さめの大文字英字+補助的に日本語。LED風のON/状態インジケータを添える
- ON/OFF・解放/未解放は彩度の差で表現する(OFFやロック中は退色)

**レイアウト**
- 上部固定ヘッダー(タイトル、メーター、音量、お題/自分の音の切替)+その下にモジュールパネルの縦積みラック
- パネルは信号の流れ順。セクションラベル(細い枠+見出し)でグルーピングする

**インタラクション**
- ノブは縦ドラッグで回す(標準的なソフトシンセ挙動)。ダブルタップでデフォルト値に戻す
- 実装時は frontend-design スキルを参照し、テンプレート的でない仕上がりを目指すこと

## 判定方式

「正解の音に近いか」の判定には2案あり、v1ではA案を採用する。

- **A. パラメータ空間距離**(v1採用): 正解パッチと現在パッチの各パラメータ差を重み付きで評価する。実装が容易で、「フィルタがまだ開きすぎ」のような具体的フィードバックを返せる。序盤の問題は許容幅を広く設定する。欠点は、異なるパラメータで聴感上同じ音になるケースを不正解にしてしまうこと
- **B. 音響特徴量距離**(将来): スペクトル重心、MFCC、エンベロープ形状などで音そのものを比較する。判定として本来あるべき姿だが、実装と閾値調整のコストが高い

**設計上の要請**: 判定器はインターフェースとして分離し、A→Bへ後から差し替え可能な構造にすること。

## 技術スタック

- **音声**: Web Audio API。減算合成はOscillatorNode / BiquadFilterNode / GainNodeの標準ノードで構成できる。AudioWorkletはB判定導入まで不要
- **フロントエンド**: React。単一ファイル指向で開始し、必要に応じて分割
- **状態管理**: パッチ=単一のJSONオブジェクトとして表現する。保存・URL共有・判定のすべてをこのJSONを介して行う
- **配布**: PWA化し、スマホのホーム画面から起動・オフライン動作可能にする。ネイティブアプリ化はしない

## 短期目標(MVP)

最小構成として「第1〜2章の10問 + 判定A + ラック型UI(実機風パネル)」を実装し、「触って学べる」体験が成立するかを検証する。エンベロープ以降の章は骨格が固まってから着手する。

MVPの完了条件:
1. スマホブラウザで音が鳴り、ノブ操作が音に即時反映される
2. お題の再生・自分の音の再生・提出・判定のループが10問分回る
3. ラック上で操作対象モジュールがハイライトされ、未解放モジュールが退色表示される
4. パネルとノブが「ビジュアルデザイン方針」の質感基準(スキュアモーフィック、署名色、画像アセット不使用)を満たす

## 長期目標・理想像

シンセ未経験の大人が全30問を完走した時点で、市販のソフトシンセの画面を見て「どこを触れば何が起こるか」が予測できる状態になっていること。卒業制作のパッチ共有機能により、学習の成果物が他者に見せられる形で残ること。

## 発展・展望

- 判定Bの導入により「別解」を許容する、より音楽的な判定へ進化させる
- FM合成・物理モデリングを扱う続編カリキュラムの可能性(物理モデリングは開発者の制御工学的関心と接続する)
- 生態系シミュレータ系の別プロジェクト構想(アーコロジー箱庭など)とのサウンド面での接続可能性

## 制約・前提条件

- 個人開発であり、開発時間は限られる。1問=1概念、v1=減算合成のみ、判定=A案という絞り込みは崩さない
- ブラウザ(特にiOS Safari)のWeb Audio制約に注意。ユーザー操作を起点としないAudioContext起動は不可
- 開発はClaude Codeで行う前提。コードは判定器・音声エンジン・UI・カリキュラムデータを疎結合に保つこと

## 未解決の問い・不確実性

- 判定Aの重み付けと許容幅の初期値をどう決めるか(実プレイでの調整が必須)
- お題の音の提示方法: 正解パッチをリアルタイム合成して鳴らすか、レンダリング済み音源を持つか(前者ならデータはパッチJSONのみで済む)
- 解説カードの文量とトーン(読み飛ばされない最小限の設計)
- 「霧笛のような」等のお題文言のセンスが体験の質を大きく左右する。カリキュラムデータは早めにプレイテストする
